# 一、基础理论

## RAG 架构原理

### 🧾 文档处理阶段（离线预处理）

1. **Local Documents**（本地文档）
   - 起始原始数据来源，可能是 PDF、Word、网页等。
2. **Unstructured Loader**
   - 使用非结构化文档加载器提取文本内容。
3. **Text**
   - 得到纯文本。
4. **Text Splitter**
   - 将长文本拆分成适合处理的小块（chunks）。
5. **Text Chunks**
   - 文本块作为后续向量化的基本单位。
6. **Embedding**
   - 使用嵌入模型（如 OpenAI Embedding、BGE、小模型等）将文本块转为向量。
7. **VectorStore**
   - 将向量存储进向量数据库（如 FAISS、Chroma、Milvus 等）以便后续检索。

### 🔍 查询处理阶段（在线推理）

1. **Query**
   - 用户输入的自然语言问题。
2. **Embedding**
   - 将 Query 转化为向量。
3. **Query Vector**
   - 得到查询向量。
4. **Vector Similarity**
   - 在向量库中查找与查询向量最相近的文本块。
5. **Related Text Chunks**
   - 检索到的相关文段。
6. **Prompt Template**
   - 定义 Prompt 模板（例如“请基于以下内容回答问题…”）。
7. **Prompt**
   - 将检索结果填入 Prompt 模板，生成最终 Prompt。
8. **LLM**
   - 调用大语言模型（如 ChatGLM）生成回答。
9. **Answer**
   1. 最终用户得到的回答。

## RAG 架构演进之路：从Naive到Advanced的改进

#### Naive RAG的问题

1. **检索质量**
   - 准确率低，导致信息不准确或产生幻觉。
   - 召回率低，可能遗漏关键信息。
   - 检索结果中包含过时或冗余的信息。
2. **生成质量**
   - 即使问题的答案未能被正确检索，仍会尝试生成答案，导致幻觉问题。
   - 答非所问、有害和偏见的回答等问题。
   - 增强过程中的挑战包括内容连贯性差、冗余重复以及过度依赖增强信息而没有额外收益。

#### Advanced RAG的改进措施

1. **数据粒度增强**
   - 修订简化数据内容，确保其准确性和可读性。
   - 删除无关信息以减少歧义，并维护上下文的一致性。
2. **优化索引结构**
   - 调整文档分块大小，以便更好地捕捉相关上下文。
   - 利用图数据索引中的节点关系实现更复杂的查询路径。
   - 构建多层索引结构以提高检索效率。
3. **元数据信息的应用**
   - 使用日期标签等元数据来强调最新信息的相关性，特别适用于时间敏感的数据检索。
4. **混合检索与对齐优化**
   - 结合多种检索策略，为每个 chunk 创建假设性问题，解决文档间的一致性问题。
5. **Hierarchical Index Retrieval 流程**
   - 通过预检索、索引总结向量、存储所有chunk的向量、选择最相关的前k个chunk，最后使用大型语言模型（LLM）生成最终答案，提高了检索和生成的质量。
