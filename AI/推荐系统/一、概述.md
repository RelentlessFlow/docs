# 一、推荐系统概述

## 推荐系统关键性指标

- Relevance：推荐的东西起码得相关才行
- Novelty：新颖的才好，推的得是人家没有的
- Serendipity：跟处对象一样一样的，机缘
- Diversity：多样性，换着花样玩才好

## 推荐系统常用词

- Item：商品，例如要从拼多多买的9.9包邮的拖鞋
- Embedding：隐向量，例如对用户商品评分矩阵进行分解
- 召回：粗略计算要返回结果，例如先从100W商品中取比较可能的100个
- 打分：要排名得有一个统一的标准；重排：最终结果排序

## 推荐系统经典流程

- 离线+近线+在线（号回+粗排+精排）
- 离线通常跑较大的模型与算法，先得到当前数据的大致结果，一定时间更新一次
- 粗排通常会跟着用户走，用户做了什么事，推荐结果也会随之更新
- 在线模块需要根据业务规则来返回最终呈现结果

#### 1. 离线（Offline）

- **作用**：训练模型、构建特征、预计算一些结果
- **特点**：**定期更新、计算量大**
- **举例**：用大数据训练深度模型（如点击预测模型、用户画像、物品embedding），每天/每小时批量生成推荐候选集
- **结果**：得出一些“用户可能感兴趣”的候选物品集合（Recall）

#### 2. 近线/粗排（Nearline / Rough Ranking）

- **作用**：进一步过滤和排序离线候选集
- **特点**：**延迟低、更新快**，可跟着用户实时行为做调整，比如点击、点赞、浏览
- **举例**：用户刚看了某本书，系统会加大类似书籍的推荐权重；某个候选内容热度突然上升，系统迅速响应
- **结果**：从离线候选中，选出几十上百个排序靠前的项目（Rough Rank）

#### 3. 在线/精排（Online / Fine Ranking）

- **作用**：实时计算、结合最新上下文做最终排序
- **特点**：**毫秒级响应**，高度个性化
- **内容**：
  - 实时获取用户状态（如刚点了A，那B优先展示）
  - 应用业务策略（如广告兜底、类目控制、内容冷启补充等）
  - 做最终多目标精排（点击率CTR、转化率CVR、多样性、用户兴趣平衡等）
- **结果**：最终决定展现给用户的顺序和内容

## 推荐系统难点与挑战

- 需要更广泛的收集用户标签并画像
- 人是善变的，随着时间的推移，兴趣也会改变
- 根据固定画像数据，推荐结果可不能固定不变
- 特征工程如何构建一直是一个大难题
- 冷启动怎么办，包括用户冷启动与商品冷启动
- 新用户来了，不知道他啥样怎么办
- 商品倒是好办，属性相对固定
- 解决方法比较多，例如直接推荐销冠

## 推荐系统技术点分析

- Embedding方向：如何更好的表示数据，肯定不用one-hot

  - one-hot 就是[0,1,0,0] 这种，有十万个用户就是十万维度，它是某个用户的某个特征的唯一标识
  - one-hot 维度太稀疏，无法表示相似性
  - one-hot 是 embedding 的前身

- 隐向量的方法在推荐中几乎无处不用，例如常见的FM及其DeepFM算法

  | 隐向量 | 用户/商品的隐藏“性格标签” | 用数字向量表示人的兴趣或物品特征       |
  | ------ | ------------------------- | -------------------------------------- |
  | FM     | 速配引擎                  | 用隐向量配对，判断两个特征是否“合得来” |
  | DeepFM | 进阶版速配 + 心理分析     | FM做快速组合配对，DNN做复杂关系建模    |

- NLP方向：如何基于文本数据来进行推荐？
  - LDA：给文本打标签，找相同“主题”的内容
  - 词向量：把文字转成“能比较的数字”，找词义相似的内容
  - 矩阵分解：用户×内容的配对系统，用行为数据学兴趣
- 知识图谱
- 深度学习，卷积网络
